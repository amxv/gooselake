# ACP Provider Implementation Plan for GG Runtime

Date: 2026-06-23

## State of Current System

GG Runtime currently supports two providers behind one `RuntimeProvider` trait:

- `ProviderKind::Codex` in `crates/runtime-core/src/provider.rs`
- `ProviderKind::Claude` in `crates/runtime-core/src/provider.rs`
- `crates/runtime-provider-codex`, a direct process-backed provider that runs the `codex` CLI per turn
- `crates/runtime-provider-claude`, a bridge-backed provider that talks to a Claude sidecar over JSON lines

The current runtime contract is terminal-result-centric:

- `RuntimeSessionManager::send_turn` allocates a runtime turn id and optionally a pre-dispatch approval id.
- The provider returns `ProviderTurnAck`.
- Runtime-core spawns `wait_for_turn`.
- The provider eventually returns one terminal `ProviderTurnResult`.
- `apply_terminal_result` persists the final turn state, appends transcript entries, and emits `turn.completed`, `turn.interrupted`, or `turn.failed`.
- Assistant transcript persistence depends on `ProviderTurnResult.usage.last_message`, `lastMessage`, `assistant_text`, or `assistantText`.

The current approval model is runtime-originated:

- If `permission_mode == "require_approval"`, runtime-core creates a pending `ApprovalRecord` before the provider proceeds.
- The provider receives `approval_id` in `ProviderSendTurnRequest`.
- Runtime calls provider `respond_approval` after the API/UI resolves that approval.

ACP differs in an important way:

- ACP `session/request_permission` requests originate from the agent during `session/prompt`.
- The ACP client must answer that request while the prompt is still active.
- This does not fit the current pre-dispatch-only approval model without either a runtime-core provider-originated approval callback or an explicit MVP limitation.

The server layer wires providers explicitly:

- Workspace members are declared in root `Cargo.toml`.
- `runtime-server` depends directly on each provider crate.
- `RuntimeServerConfig.providers` has `codex`, `claude`, and `claude_auth_mode`.
- `bootstrap_runtime` constructs Codex and Claude providers, registers enabled providers, then runs startup recovery.
- HTTP provider routes are partly generic (`/v1/providers`, `/v1/providers/{provider}/models`) and partly provider-specific (`/v1/providers/codex/auth/status`, `/v1/providers/claude/auth/*`).
- Diagnostics routes iterate provider registry metadata and call `provider_auth_status`.
- OpenAPI is generated by source parsing in `crates/runtime-server/src/openapi.rs`, with hardcoded summaries and request-body recognition.

Team, process, worktree, and MCP flows are provider-agnostic:

- Team spawn accepts a provider string and parses it with `ProviderKind::from_str`.
- Spawned sessions inherit or receive a model, cwd, permission mode, and metadata through `CreateSessionInput`.
- Worktree assignment is runtime-owned and supplies the provider session `cwd`.
- GG MCP tools route back through `/v1/mcp/invoke` with a required caller session id.
- Claude already proves a provider can inject the GG MCP sidecar into sessions.

ACP research conclusions that constrain implementation:

- Stable ACP v1 transport today is stdio.
- Streamable HTTP is still draft-level and should not be the initial GG integration target.
- ACP should be added as a third provider adapter, not as a replacement for GG's runtime contract.
- The official Rust SDK (`agent-client-protocol` plus Tokio helpers) is the preferred implementation path.
- ACP model discovery is usually represented as session config options, not as a provider-global model list.
- ACP config options are session-scoped and the current GG HTTP API has no matching session-config endpoint.

## State of Ideal System

ACP is available as a third provider alongside Codex and Claude with these properties:

- `ProviderKind::Acp` serializes as `"acp"` and is accepted wherever the runtime accepts provider names.
- A new `runtime-provider-acp` crate implements `RuntimeProvider`.
- Runtime-server config can enable or disable ACP independently.
- Bootstrap constructs and registers ACP only when enabled.
- `/v1/providers` includes ACP metadata when enabled.
- `/v1/providers/acp/auth/status` reports meaningful ACP agent auth readiness without pretending all ACP agents support API key or JSON auth import.
- `/v1/providers/acp/auth/logout` exists only if the implemented ACP adapter supports the ACP `logout` method for the configured agent. Otherwise the plan should intentionally omit it or return unsupported only behind an approved route decision.
- `/v1/providers/acp/models` works without misleading clients. It either returns models discovered from ACP session config options when available, or an empty list with documented semantics.
- ACP session create/resume/close maps to ACP `initialize`, optional `authenticate`, `session/new`, `session/resume` or `session/load`, and `session/close`.
- ACP turns map to `session/prompt`, `session/update`, `session/prompt` response, `session/cancel`, and final `ProviderTurnResult`.
- Final assistant text is always merged into `ProviderTurnResult.usage.last_message` or `assistant_text` so existing transcript persistence works.
- ACP stop reasons map deterministically into `ProviderTurnStatus`.
- GG-owned worktree/process/team/MCP flows keep working because ACP receives the runtime-selected `cwd` and the GG MCP server definition at session setup.
- Startup recovery has explicit behavior for ACP sessions and active turns.
- Tests cover the provider contract with a fake ACP agent, HTTP/provider route integration, docs/OpenAPI sync, and mixed provider/team/worktree flows.

## Decisions Needed Before Implementation

These decisions need user or lead approval before implementation starts:

1. **ACP agent launch contract.**
   Decide the user-facing config shape for the ACP agent command. Recommended: `providers.acp.command`, `providers.acp.args`, `providers.acp.env`, `providers.acp.transport = "stdio"` with stdio as the only supported initial transport.

2. **Default ACP enabled state.**
   Recommended: default `providers.acp.enabled = false` until a default ACP agent command is chosen. This avoids boot failures on machines without an ACP agent installed.

3. **Provider-originated approval support.**
   Choose one:
   - Recommended complete path: add a runtime-core provider-originated approval API used by ACP `session/request_permission`.
   - Temporary MVP path: configure ACP sessions to avoid agent-side permission requests and fail fast if one arrives. This is easier, but it does not satisfy full ACP tool-call lifecycle support.

4. **ACP auth HTTP surface.**
   Recommended initial route: `GET /v1/providers/acp/auth/status`.
   Add `POST /v1/providers/acp/auth/logout` only if the ACP SDK/agent integration can reliably expose `logout`.
   Do not add ACP `api-key`, `import-json`, or `import-file` routes unless a specific configured ACP agent supports them.

5. **Model list semantics.**
   Decide whether an empty ACP model list is acceptable when no provider-global model catalog exists. Recommended: allow empty list and document that ACP model selection may be agent/session-config-specific.

6. **Startup recovery semantics for active ACP turns.**
   Recommended: resume session context when possible, but mark in-flight turns failed/interrupted if the ACP agent cannot resume an active `session/prompt` after runtime restart.

7. **Incremental event fidelity.**
   Decide whether ACP `session/update` notifications should be persisted as first-class runtime events in this implementation. Recommended initial scope: collect updates provider-side, return terminal result, and persist only final assistant text/tool summary unless provider-originated approvals require runtime events.

## Cross-provider Requirements

- Keep `RuntimeProvider` as the stable abstraction. ACP should not leak protocol-specific types into HTTP handlers or team/worktree services.
- `ProviderKind::from_str` must parse `"acp"` anywhere the runtime accepts provider strings, including team spawn and session creation.
- Provider metadata should remain enough for `/v1/providers`, diagnostics, and health checks.
- Provider terminal results must follow runtime-core invariants:
  - `runtime_session_id` and `turn_id` must match runtime ids.
  - terminal status must be one of `completed`, `interrupted`, or `failed`.
  - assistant text must be present in `usage.last_message` or `usage.assistant_text` when available.
  - duplicate terminal results must be idempotent only when status matches.
- Provider auth methods may remain unsupported unless explicitly backed by ACP protocol behavior.
- Runtime-owned services stay runtime-owned:
  - Worktree creation and assignment stays in `RuntimeWorktreeService`.
  - Background processes stay in `RuntimeProcessManager`.
  - Team messaging stays in `RuntimeTeamCommsService`.
  - GG MCP sidecar remains the tool bridge ACP agents should consume.
- Avoid adding ACP terminal emulation as a second process-control system unless explicitly approved later.

  - `crates/runtime-provider-acp/src/mapping.rs`
  - `crates/runtime-provider-acp/src/tests/fake_agent.rs` or inline test fixture

#### Concrete work to do

- Implement a subprocess-backed ACP client using stdio.
- Use the official Rust SDK if it supports the needed client role APIs. Otherwise implement a small typed JSON-RPC stdio client against official ACP schema types.
- Add provider state shaped similarly to Claude:
  - live ACP connection/process handles
  - runtime session id to ACP session id map
  - active turn map
  - completed turn map
  - waiters or wait request path
  - request id map if using raw JSON-RPC
  - shutdown/cancellation state
- Implement connection initialization:
  - spawn configured command and args
  - wire stdin/stdout/stderr
  - send `initialize`
  - handle protocol version and capabilities
  - handle optional `authenticate` if required by agent and approved auth mode supports it
- Implement `auth_status`:
  - report disabled provider when disabled
  - report command missing or not configured
  - report whether initialization/auth negotiation succeeds if lightweight enough
  - return `mode` such as `acp_agent`, `not_required`, `agent_managed`, or `unsupported` as approved
- Implement `create_session`:
  - ensure provider enabled
  - ensure ACP connection initialized
  - send `session/new` with `cwd`
  - include `mcpServers` or ACP SDK equivalent for GG MCP sidecar if enabled
  - map runtime `model` into ACP session config only if the agent exposes a model config option or an approved config mapping exists
  - store ACP `sessionId` as `provider_session_ref`
- Implement `resume_session`:
  - use ACP `session/resume` when supported
  - if only `session/load` is available, decide whether to replay/load and then return a session handle
  - preserve `provider_session_ref` and canonical ref behavior
  - fail clearly when the agent cannot resume and runtime recovery should mark the session failed
- Implement `send_turn`:
  - flatten GG `input: Vec<Value>` into ACP prompt content using a deliberate mapping:
    - text entries become text content
    - unsupported multimodal entries are either stringified with type labels or rejected, according to existing Codex behavior and approved scope
  - call ACP `session/prompt`
  - process `session/update` notifications while the prompt is active
  - return `ProviderTurnAck` promptly once the prompt request is accepted or tracked
- Implement `wait_for_turn`:
  - wait for the active `session/prompt` response
  - combine accumulated assistant text chunks and terminal result data
  - map ACP stop reasons:
    - normal end turn -> `Completed`
    - cancelled -> `Interrupted`
    - refusal, max tokens, max turn requests, protocol error -> `Failed` unless approved otherwise
  - merge final assistant text into `usage.last_message` and `usage.assistant_text`
  - include raw ACP usage/update metadata under namespaced fields such as `acp` to avoid colliding with current extraction keys
- Implement `interrupt_turn`:
  - send ACP `session/cancel`
  - mark local cancellation requested
  - return after the cancel request is accepted, leaving terminal reconciliation to `wait_for_turn`
- Implement `close_session`:
  - send ACP `session/close` when supported
  - tear down local handles
  - terminate child process only when no sessions remain on that connection or per configured instance lifecycle
- Implement error mapping:
  - ACP session/turn not found -> `RuntimeError::NotFound`
  - invalid ACP state -> `RuntimeError::InvalidState`
  - malformed/missing required ACP fields -> `RuntimeError::ProtocolViolation`
  - subprocess IO/death -> `RuntimeError::Io`
  - disabled/misconfigured command -> `RuntimeError::Configuration` or `Bootstrap`

#### Acceptance criteria

- A fake ACP agent test can create, resume, send, wait, interrupt, and close a session.
- Completed ACP turns persist assistant text through existing runtime transcript logic.
- ACP process failure produces a failed provider result or clear runtime error without hanging waiters.
- Interrupt uses ACP cancellation and returns an interrupted or failed terminal state deterministically.
- Unsupported ACP update types do not panic or corrupt provider state.

#### Validation/check commands

- `cargo test -p runtime-provider-acp`
- `cargo test -p runtime-provider-acp real_adapter_contract`
- `cargo test -p runtime-core terminal`
- `cargo test -p runtime-core startup_recovery`
- `cargo check --workspace`

#### Risks and fallback notes

- ACP SDK client APIs may not expose all required lower-level hooks. Fallback: use official schema types and raw JSON-RPC over stdio.
- Some ACP agents may not support `session/resume`. Implement explicit capability detection and recovery behavior rather than silently pretending resume worked.
- If `session/prompt` is a long-running request, provider implementation must avoid blocking `send_turn` indefinitely. It should track the in-flight request and let `wait_for_turn` await the terminal response.
- If stdout notifications and request responses interleave, message dispatch must be single-reader and fan out by request/session ids.

#### Concrete work to do

Recommended complete path:

- Add a runtime-core method that providers can use to register a provider-originated approval during an active turn.
- Keep the method on `RuntimeSessionManager`, not on HTTP.
- Suggested input fields:
  - `runtime_session_id`
  - `turn_id`
  - `provider_approval_ref`
  - `tool_call_id`
  - `request: Value`
- Suggested behavior:
  - validate session and active turn
  - allocate a GG approval id
  - persist `ApprovalRecord`
  - set turn/session status to `waiting_for_approval` only if appropriate
  - emit `approval.requested`
  - return the GG approval id plus a future/receiver or lookup path the provider can wait on
- Add an internal provider approval response notification path:
  - Current `respond_approval` calls provider `respond_approval` after resolving.
  - ACP provider can use that existing method if it records a mapping from GG approval id to ACP permission request id and waits for the provider method to be called.
- Update ACP provider:
  - On ACP `session/request_permission`, create a runtime approval.
  - Suspend the ACP permission RPC response until `respond_approval` is called.
  - Map GG `accept` to the approved ACP option.
  - Map GG `decline` to ACP cancellation/rejection.
  - Handle timeout/runtime shutdown by rejecting or cancelling the ACP permission request.
- Update startup recovery:
  - preserve provider-originated pending approvals the same way existing `waiting_for_approval` turns are preserved.
  - define what happens when the runtime restarts while an ACP permission request is waiting. Recommended: pending approval remains visible, but the in-flight ACP turn is marked failed unless the ACP agent can resume the prompt and permission request.

Temporary MVP path if full approval support is not approved:

- In ACP provider, if `session/request_permission` arrives, fail the active turn with a clear unsupported error.
- Document that ACP agents must be configured not to request permissions for the initial release.
- Add tests proving the failure is terminal and does not hang.

#### Acceptance criteria

- Complete path:
  - ACP permission requests create normal GG `ApprovalRecord`s.
  - `/v1/sessions/{session_id}/approvals/{approval_id}` resolves ACP permission requests.
  - Accept and decline both unblock the ACP turn.
  - Pending provider-originated approvals survive runtime state hydration consistently.
- MVP path:
  - ACP permission requests fail the turn clearly.
  - No orphaned active turn, waiter, or approval remains.

#### Validation/check commands

- `cargo test -p runtime-core approval`
- `cargo test -p runtime-core startup_recovery`
- `cargo test -p runtime-provider-acp approval`
- `cargo test -p runtime-server approval`
- `cargo test -p runtime-server diagnostics`

#### Risks and fallback notes

- This is the highest-risk phase. The current runtime approval model was built for runtime-originated pre-dispatch approvals.
- If adding provider callbacks creates circular dependencies, keep the callback as an interface passed into `AcpProvider` at construction time rather than making providers own `RuntimeSessionManager`.
- If provider-originated approvals require store schema changes, keep schema additive and verify SQLite hydration tests.

## Minimal Non-negotiable Test Strategy

Do not ship ACP with only compile checks. Minimum deterministic coverage should include:

- `ProviderKind::Acp` parse/serialize behavior.
- ACP provider metadata, healthcheck, auth status, and model list.
- ACP fake agent lifecycle: initialize, session/new, session/resume, session/prompt, session/cancel, session/close.
- Terminal result mapping and assistant text persistence into `usage.last_message`.
- Provider failure and malformed protocol response handling.
- Startup recovery behavior for ACP sessions.
- HTTP provider list/auth/models/diagnostics coverage.
- Session create/send/events coverage over HTTP.
- Mixed-provider team messaging with ACP included.
- Spawned ACP member with created and reused worktree.
- GG MCP process tool invocation with ACP session caller identity.
- API docs sync checks if routes change.
