# Gooseweb acceptance contract

This directory is the source-controlled contract for Golden Goose migration browser review. `schemas/` versions reusable P01-P56 manifest, active-registry, evidence, clearance, non-clearance outcome, immutable phase-graph seed, and external lifecycle-attestation shapes. `manifests/` contains only production phase scenarios; `manifest-registry.json` is candidate intent and names exactly one authoritative active manifest tuple for each phase eligible for review; `ledger/phase-graph-seed.json` freezes the P00-P56 dependency graph and the migration's leases 1-5 bootstrap checkpoint; and `allowlists/` defines the reproducible browser capture boundary. The tracked seed is deliberately not current lifecycle state and is never changed to record a review of its own tree. Validator fixtures live outside `manifests/` and cannot be referenced by evidence or clearance. Published schemas restrict paths to nonempty safe segments beneath `verification/gooseweb/manifests/`, while containment checks provide defense in depth. Manifest schema revision is exactly v4; manifest document revisions and phase bases are positive/exact phase-specific values. Every evidence/outcome record binds its phase and manifest to the active registry tuple, active file, and exact candidate-commit blob. The deterministic validator applies every schema to its document, then enforces cross-record, lifecycle, phase-specific base/range/head Git relationships, capture, and evidence-file rules that JSON Schema cannot express. It runs in the existing Gooseweb test entrypoint.

Evidence belongs at `tmp/gg/gooseweb-migration/<phase-id>/<sha7>/attempt-<n>/` and must match `schemas/evidence-run.schema.json`. It is intentionally not committed. Each run contains a manifest copy and SHA-256, environment descriptor, screenshots, console/network/WebSocket results, redacted runtime/Tower/store state, check results, applicable metrics, report, and exact-head clearance JSON.

Never retain credentials, cookies, CSRF values, bearer tokens, realtime tickets or query secrets, provider authentication, raw image bytes, secret configuration, or private keys. Redact at capture time; a later cleanup is not an acceptable control. The validator resolves each referenced artifact inside the declared root, requires every file, fully parses PNG chunks/CRC/zlib payload and dimensions, verifies the manifest copy hash, scans all text/JSON artifacts, and compares browser captures to the exact allowlists. Unavailable WebSocket capture is never inferred: its ID must resolve exactly once to the validated manifest's scenario baseline.

Console evidence is the complete unfiltered `agent-browser console` multiset captured after `document.readyState === "complete"` plus one second. It must equal exactly one finite, source-controlled benign development variant. Emission is timing/cache-dependent among three individually observed literals—Vite connecting, Vite connected, and the exact React DevTools notice—so v6 enumerates their complete eight-member power set, including empty, as eight literal exact multisets. No regex, filtering, normalization, suppression, or open-ended informational allowance is applied. Every variant and capture contains zero warnings, errors, or exceptions, and any such message fails unconditionally. Network evidence retains the complete unfiltered `agent-browser network requests` list with query keys but no query values. The validator—not the reviewer—may omit only same-origin 2xx/3xx stylesheet, font, script, and module resources by exact resource type. Network allowlist v5 preserves the P01/P02 v3 aggregate contract and adds the P03 v4 reconstruction contract: `initial_load`, `ordinary_reload`, and `fresh_context` are retained in exact order, every raw request is attributed monotonically to one segment, and every segment contains exactly one successful same-origin query-free document plus dev-ticket request. Ordinary reload is recorded honestly and is never labeled cache-bypassing or hard reload. Fresh-context evidence separately proves a second unique fresh-ephemeral session/profile had no prior browser cache, IndexedDB, or session-storage state before its explicitly exercised login flow. Each segment may additionally retain at most one favicon 404 mapped to `BASE-P01-FAVICON-NOT-FOUND`. Cache/context absence never fabricates the favicon request; presence never suppresses or approves its failure. No other 4xx/5xx, cross-origin, document, API, WebSocket, or unclassified request can be omitted or accepted, and every retained failure must resolve exactly once to the manifest baseline register.

Every attempt ends with exactly one typed review outcome. Descriptor, environment, and outcome bind the same attempt, reviewer, browser session/tuple, lease lifecycle, stack configuration, base/range/head/tree/served tuple, clean/hot state, and manifest. `cleared` references an exact-head clearance issued only after stop, cleanup, and lease release. `changes_required` references a Git-verified schema-valid non-clearance record with `clearance_issued: false`, `product_approved: false`, no approval recipient, and at least one stable finding ID. A failed attempt never fabricates a clearance.

Post-release authority is the external store at `tmp/gg/gooseweb-migration/lifecycle/`, not a tracked-file update. Immutable records live at `attestations/gooseweb-lifecycle-<sequence>-<raw-byte-sha256>.json`; `current.json`, validated by `schemas/lifecycle-current.schema.json`, identifies the one canonical tip. Each v2 attestation declares a safe predecessor path and exact raw-byte SHA-256, binds the immutable phase-graph seed blob from Git, and hashes the complete evidence descriptor and typed outcome. Validation scans every stored file rather than trusting a caller object: it requires exactly one genesis, loads and schema-validates every predecessor from disk, rejects missing/malformed/hash-mismatched predecessors, cycles, forks, disconnected files, replacement genesis records, removed/reordered attempt prefixes, backward generation timestamps, and a current pointer that is not the unique tip. Every node—not only the tip—independently revalidates all cumulative evidence/outcome hashes and tuples, the combined lease chain, outcome terminal-time bounds, and its claimed effective states and eligible phases. A later corrected tip can never hide forged historical authority. Clearance can therefore make P01 authoritative `cleared` and P02 eligible after release without changing or re-reviewing the cleared code tree. Missing or invalid current lease/clearance evidence leaves P01 uncleared and cannot advance P02.

Baseline registers contain zero or more entries and clearance must equal the complete candidate-manifest register. P01-P05 may clear verification-infrastructure scope with finite mapped baselines, no product approval, and a manifest that remains pending or honestly blocked. P06-P55 product clearance requires an empty baseline register, product approval, and an exact candidate manifest whose `scenario.product_clearance` is `approved`; P56 requires the same approved manifest state with an empty register and integration-release scope. `known_defects` is always empty at clearance.

Browser acceptance uses one fresh, uniquely named, headless, real-Chromium `agent-browser` session. The descriptor records the exact Chromium binary/version and fresh-profile policy. Headed evidence is rejected, and a browser mode, binary, version, profile, reviewer, lease, manifest, stack, head, tree, or served-head change invalidates clearance. Only the supervisor may own the migration-wide stack lease and lifecycle. Reviewers are read-only and own only browser interaction and evidence.

## P03 headless browser evidence

P03 makes the installed `agent-browser` CLI plus the user's real local Google Chrome/Chromium the sole browser acceptance and evidence substrate. The reviewer procedure is `procedures/p03-headless-agent-browser.md`; it is intentionally prose and command vocabulary, not a committed runner. Each P03 standard evidence-run descriptor must name the fixed `p03-browser-evidence.json` artifact, validated against `schemas/p03-browser-evidence.schema.json`; missing or tuple-mismatched artifacts reject exact-head evidence. The validator cross-binds phase/attempt/head/tree, browser session/local executable/full binary version/mode/profile, an explicitly reduced `HeadlessChrome/<same-major>.0.0.0` token, typed `navigator.webdriver`, and supplemental exact high-entropy Chrome full-version brands when available. API-unavailable fallback requires an explicit reason and cannot weaken the local-binary/reduced-UA-major/headless tuple. It also binds supervisor lease/stack ownership, screenshot/viewport/composer geometry, manifest baselines/known defects, and procedure-overhead completion; requires the exact manifest-linked roster/composer journey, canonical `P02 deterministic turn action` submitted text, and command cardinality; parses typed runtime/Tower/Worker-store authority JSON and cross-checks its real correlation/content/identity/cursor/cardinality/discrepancy fields and the manifest-text hash; retains React screenshot evidence; permits measured downstream divergence cascades while requiring the declared first divergence to be the earliest fixed-order divergent layer; requires typed reload/reconnect/navigation/fresh-context integrity and initial-null context nonces; and enforces complete IndexedDB/cookie/local/session/CacheStorage/service-worker disposal. The contract binds explicit `--headed false`, a globally unique named ephemeral session, complete console/network/WebSocket capture, capture-time redaction, default/production fixture leakage, and wrong-head/stale-context checks.

The P03 smoke self-test preserves the original console, network, WebSocket, wrong-head, wrong-viewport, fixture-leak, stale-context, headed-mode, and non-real-Chromium seeds and additionally rejects missing/false/mismatched journey, cardinality, authority, divergence, reconstruction, automation, nonce, CacheStorage, service-worker, standard-evidence linkage, and reconstruction-network segment proofs. Its positive segmented capture includes all three required loads plus retained static traffic and a finite mapped favicon; negative cases cover missing/extra/duplicate/reordered segments, filtering, cross-origin/status/query errors, missing/extra requests, raw-count forgery, repeated favicon, wrong trigger/context, incomplete boundaries, and segment misattribution. Typed capability evidence rejects ordinary-reload command success without navigation, traffic, or semantic reconstruction and separately rejects a reused/nonempty fresh context or non-login prior state. It also verifies the P03 manifest keeps `known_defects: []` and the exact ten finite P01/P02 baseline entries. P03 installs no package, browser binary/cache/config, CI browser job, alternate framework, or browser runner; the supervisor continues to own every stack process and port.

## P02 deterministic source

P02 adds `p02-fake-gooselake/v1`, a bounded in-memory implementation of the existing public runtime JSON and SSE contracts. It is verification infrastructure in `goosetower::verification`; production runtime and gateway configuration never select it implicitly. The lease-holding supervisor may start it explicitly with `cargo run -p goosetower --example p02_fake_gooselake` and point the real Tower source configuration at `http://127.0.0.1:18102`. Implementers and reviewers do not start it.

Fault injection and the fake/runtime observer are available only below `/__verification/v1/` and return 404 unless `x-gooseweb-verification-control: p02-local-control` is supplied. Supported typed controls are `emit_next`, `emit_batch`, `delay_next`, `duplicate_next`, `gap_next`, `change_epoch`, `disconnect_next`, `offline`, `delay_terminal`, `reject_next_command`, and `reset`. Batches cap at 32, replay pages cap at 2,000, observer collections are finite, clock/IDs are fixed, and redaction happens before observer values leave the fake source. These controls are absent from production runtime, Tower, and Gooseweb routes.

The complementary Gooseweb observer smoke imports the production Protobuf decoder and external store. It records bounded redacted frame/store snapshots and rejects missing, skipped, reordered, inferred-unavailable, or wrongly attributed evidence. Browser-visible checks at 1440×1000, 820×1000, and 520×900 remain supervisor/reviewer work under the migration lease; P02 source tests do not create a second browser harness.

The fake implements the exact paths used by `GooselakeRuntimeClient`, including plural team message/broadcast routes, provider model/auth, session/team/process global and scoped replay/stream, processes, worktrees, and diagnostics. JSON replay defaults to 500 and caps at 10,000. SSE replay defaults to 2,000, clamps to 1–10,000, validates `Last-Event-ID` even when a query cursor exists, and uses production precedence `after_seq.or(last_event_id)`. Ordinary global and scoped streams mirror production's 10-second keepalive; the explicit `disconnect_next` fault remains a finite stream and is never masked by keepalive. Scoped replay compares scoped `seq`; global replay compares `row_id`. The integration check drives one command through the authenticated runtime client, open replay/live SSE fan-in, real materializer and gateway frame encoder, production `RealtimeWorkerCore`, production worker-output reducer, and external store; its consumer refuses a fabricated or missing frame argument.

P02 also directly observes `BASE-P02-SSE-TOTAL-DEADLINE-RECONNECT-CURSOR-LOSS`: the production `GooselakeRuntimeClient` applies its 30-second total JSON request deadline to the infinite SSE request, so reqwest terminates the response body even though keepalives arrive. `RuntimeSseFanIn` then reconnects from its earlier outer cursor after already deduplicating replayed rows. P02 does not override or correct that P06 product behavior; normal browser terminal success remains blocked until P06 owns the request-deadline/cursor-continuity repair.

For a leased P02 review stack, the supervisor runs the feature-gated `p02_observed_goosetower` example instead of the normal Tower binary, sets `P02_TOWER_OBSERVER_ADDRESS` to a fourth isolated loopback port, supplies `P02_TOWER_OBSERVER_SECRET`, and sets `VITE_GOOSEWEB_DEBUG_FRAMES=true`. The example serves the unchanged product router on the configured Tower port and a separate secret-gated verification router on the observer port. The reviewer captures actual served Tower layers from `GET /__verification/v1/tower/materializer` and `GET /__verification/v1/tower/frames` with `x-gooseweb-verification-control`; both collections are bounded and redact secrets before values leave the verification process. These routes, capture fields, and socket instrumentation are absent from the default feature set and normal `gg-goosetower` binary. The live browser Worker/store layer is read-only at `window.__GOOSEWEB_VERIFICATION_OBSERVER__.capture()`. It exists only in an explicit development debug-frames build, retains at most 128 capture-time-redacted frame/output summaries, and is absent from production and default non-debug builds. A terminal command is compared across the fake observer, materializer `recent_ledger`/`session_details`, served frames, and browser store. Any live absence requires a fresh exact-layer observation and a new manifest mapping before it can be classified; missing evidence is a failure and may not be inferred from a later layer or an older attempt.

Epoch reconstruction is supervisor-owned and never mutates an active reviewer stack. Attempt A runs the source with `P02_FAKE_SOURCE_EPOCH_NUMBER=1` and Tower `source_epoch=p02-epoch-001`; after evidence, the supervisor stops, cleans, and releases that lease. Attempt B uses a fresh `DEV_DIR` and lease, starts the same exact code with `P02_FAKE_SOURCE_EPOCH_NUMBER=2` and Tower `source_epoch=p02-epoch-002`, then the reviewer verifies reset source sequence IDs reconstruct without old-epoch dedupe. Both attempts are immutable, independently described, and never overlap. The in-process epoch test uses the same paired source/Tower configuration contract; `change_epoch` remains a fault signal, not a claim that a running Tower can change its configured epoch.
